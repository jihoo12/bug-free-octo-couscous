02 File format and statements

The CLI reads your source file line by line. Each non-blank, non-comment line is one statement. Statements can define names, check terms, or infer the type of a bare term. Definitions persist — every line sees the names defined on all previous lines.

Statement kinds

def x : T = e — define with explicit type

Checks that e has type T, then binds the name x to e in all subsequent lines. The type annotation is mandatory when the term cannot be inferred on its own (e.g. a bare lambda).

def myBool : U0 = U0
def id : Π(x:U0). U0 = λx. x

def x = e — define with inferred type

Infers the type of e, then binds x. Use this when the type is obvious.

def U = U0          -- x = U0, type inferred as U1
def idId = id id    -- uses previously defined id

check label : T = e — check without binding

Verifies e : T and reports success or failure, but does not add anything to the environment. Useful for assertions and tests.

check myLemma : Path U1 U0 U0 = ua (mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b))

<term> — bare term (infer only)

A line that is none of the above is treated as a bare term whose type is inferred and printed. Nothing is added to the environment.

U0                          -- infers U1
Π(A:U0). Π(x:A). A         -- infers U1

Comments and blank lines

-- This is a comment and is ignored entirely.

-- Blank lines are also ignored.

def A = U0          -- definitions accumulate across lines
def B = U0
Equiv A B           -- A and B are in scope here

Scoping rules

  • Names are in scope from the line after the def that introduces them.
  • A def on line n can reference any name defined on lines 1 … n−1.
  • check and bare terms have read-only access to the environment.
  • There is no shadowing protection — redefining a name silently replaces it.

Why not one term per line?

The statement format lets you define helper types and functions once and reuse them. The old style required everything inline:

-- Old style — everything inline, hard to read:
transport (ua (mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b))) U0

-- New style — build up in named steps:
def idEquiv = mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b)
def uaPath  = ua idEquiv
transport uaPath U0

03 Lexical basics

Identifiers (names)

A name starts with a letter or underscore, then continues with letters, digits, underscores, or single-quotes.

x       -- valid
myVar   -- valid
x'      -- valid (prime is allowed after the first char)
_foo    -- valid
i0      -- NOT a name — it is an interval literal
U2      -- NOT a name — it is a universe

Keywords

The following words are reserved and cannot be used as variable names:

def  check
Path  hcomp  Glue  glue  unglue  Equiv  mkEquiv  equivFwd  ua  transport

A keyword is only recognised when it is not immediately followed by a letter, digit, or underscore. So Glue is the keyword, but GlueExtra would be parsed as a plain name. Likewise uaβ would be a plain name, not the ua keyword.

Whitespace

Whitespace (spaces, tabs, newlines within a token stream) is ignored everywhere between tokens. You can add as much spacing as you like for readability.

05 Interval expressions

The interval 𝕀 is the unit interval of cubical type theory. Interval expressions appear inside […] brackets in hcomp, Glue, glue, and unglue. They have their own mini-grammar, separate from the term grammar.

Atoms

0           -- the bottom endpoint (falsy face)
1           -- the top endpoint (truthy face)
i0  i1  i2  -- named interval variables (i followed by one or more digits)
( expr )    -- parenthesised sub-expression

Operators — precedence low → high

OperatorSymbolMeaningAssociativity
Join∨logical or / union of facesleft
Meet∧logical and / intersection of facesleft
Negation¬flip the faceright (prefix)

So ¬i0 ∧ i1 ∨ i2 parses as ((¬i0) ∧ i1) ∨ i2.

Examples

[i0]            -- the bottom face  → hcomp reduces to the base
[i1]            -- the top face     → hcomp reduces to tube@1
[i0 ∨ i1]      -- union of two faces
[i0 ∧ ¬i1]     -- intersection with a negated face
hcomp U1 [i0] ⟨i⟩ U0 U0   -- phi = 0, so hcomp returns the cap U0
hcomp U1 [i1] ⟨i⟩ U0 U0   -- phi = 1, so hcomp returns tube@1 = U0
Glue  U0 [i0] U0            -- phi = 0, so Glue reduces to the base type U0
Glue  U0 [i1] U0            -- phi = 1, so Glue reduces to the fibre U0

06 Term grammar — the big picture

Here is the full grammar. Sections below explain each piece. te means: ( term ) or atom — parentheses required for compound terms.

-- Top-level statements stmt ::= def <name> : term = term | def <name> = term | check <name> : term = term | term -- Terms term ::= λx. term -- lambda abstraction | ⟨x⟩ term -- path abstraction | Π(x : term). term -- dependent product | app app ::= atom ( @ atom -- path application (left-assoc) | atom -- function application (left-assoc) )* atom ::= U<n> -- universe | 𝕀 -- interval pseudo-type | i<n> | 0 | 1 -- interval literals | Path atom atom atom -- path type | hcomp atom [iexpr] u atom -- Kan composition | Glue atom [iexpr] te -- Glue type | glue [iexpr] te atom -- glue element | unglue [iexpr] te atom -- unglue element | Equiv atom atom -- equivalence type | mkEquiv atom atom te te te te -- build equivalence | equivFwd te atom -- apply forward map | ua te -- univalence map | transport te atom -- transport along a path | <name> -- variable | ( term ) -- parenthesised term

The key insight is the two-level split: binders and application live at the term level; the individual pieces of an application spine (function, arguments) are atoms. This controls which forms need parentheses when nested.

07 Binders — λ, Π, ⟨⟩

Lambda abstraction — λx. body

λ followed by a name, then ., then the body (which is a full term). The body extends as far right as possible — λx. λy. x is λx. (λy. x).

λA. λx. x          -- identity function (taking type then value)
λf. λg. λx. f (g x) -- function composition

Dependent product — Π(x:A). B

Π followed by a binder in parentheses (x : A), then ., then the result type B. Binders nest naturally:

Π(A:U0). Π(x:A). A
Π(A:U0). Π(B:U0). Π(C:U0). Π(f:Π(x:B).C). Π(g:Π(x:A).B). Π(x:A). C
Π(f:Π(x:U0).U0). U1
Π(p:Path U1 U0 U0). U1
Π(e:Equiv U0 U0). Path U1 U0 U0

Path abstraction — ⟨i⟩ body

The angle brackets ⟨⟩ (U+27E8 / U+27E9) contain the interval variable name. The body is a full term. Used to introduce an element of a Path type.

⟨i⟩ U0              -- constant path on U0

08 Application and path application

Function application

Just juxtapose terms, left-associatively:

f x             -- f applied to x
f x y           -- (f x) applied to y
f (g x)         -- f applied to (g x) — parentheses required here

Path application — t @ r

The @ operator applies a path term to an interval expression:

p @ i0          -- evaluate path p at the bottom endpoint
p @ i1          -- evaluate path p at the top endpoint
p @ r           -- evaluate path p at interval variable r

Both @ and plain application are left-associative and at the same precedence level. They can be mixed freely in a spine:

f x @ i0 y      -- parsed as  ((f x) @ i0) y

09 Atoms

Universes — U<n>

U0    U1    U2    U3    ...

U0
U1
U2
Path U2 U1 U1      -- path between two U1's, living in U2

U0 is the smallest universe; U1 : U2, U0 : U1, etc.

Interval pseudo-type — 𝕀

The type of interval expressions. It is not a universe element — it is a pseudo-type that can only appear in certain positions.

Interval literals

0       -- the bottom endpoint I0
1       -- the top endpoint   I1
i0  i1  i2  ...   -- interval variables

These are only recognised as interval literals when not followed by more identifier characters. So i0 is a literal, but i0foo would start a name lookup for i0foo.

Variables

A name resolves to a value at parse time, checking first against local binders (lambda, Pi, path abstraction), then against the global environment from preceding def statements. If a name is not in scope, the parser fails with unbound variable: <name>.

Π(A:U0). Π(x:A). A     -- the final A refers to the outer Π binder

def Nat = U0            -- Nat is now a global name
Π(n:Nat). Nat           -- Nat resolves to U0 from the global env

10 The parenthesisation rule

This is the most important practical rule for avoiding parse errors.

Path, hcomp, Glue, glue, unglue, Equiv, mkEquiv, equivFwd, ua, and transport each take multiple arguments. They can appear at the head of an application spine without parentheses, but when one of them appears as an argument to something else, you must parenthesise it.

Path U1 U0 U0                                  -- ✅ Path is the head
Path U2 (Path U1 U0 U0) (Path U1 U0 U0)       -- ✅ compound Path as argument, wrapped
Π(p:Path U1 U0 U0). U1                         -- ✅ Path type as domain, parenthesised
Path U2 Path U1 U0 U0 Path U1 U0 U0            -- ❌ Would fail — parser reads Path U2 Path U1 then fails
Equiv U0 U0                                     -- ✅ Equiv as head
Π(e : (Equiv U0 U0)). Path U1 U0 U0            -- ✅ Equiv as argument, parenthesised
ua (mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b))  -- ✅ ua as head
transport (ua (mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b))) U0  -- ✅ ua result as argument

Inside parentheses, the full term grammar is active again, so you can nest arbitrarily deep.

11 Cubical constructs in detail

Path A u v — Path type

All three arguments are atoms (parenthesise if compound). The resulting type is the type of paths in A from u to v.

Path U1 U0 U0
Path U2 U1 U1
Path U2 (Path U1 U0 U0) (Path U1 U0 U0)   -- path of paths
Π(A:U1). Π(B:U1). Path U1 A A

hcomp A [φ] u u0 — Kan composition

ArgumentSyntaxWhat it is
AatomThe type to compose in
[φ][ interval expr ]The constraint face
ubare ⟨i⟩ atom, ( term ), or atomThe tube — a path open on one side
u0atomThe cap — the base element
 
hcomp U1 [i0] ⟨i⟩ U0 U0     -- phi=0: reduces to cap U0
hcomp U1 [i1] ⟨i⟩ U0 U0     -- phi=1: reduces to tube@1 = U0

β-reduction: [i0] → cap u0; [i1] → u @ 1 (tube at 1).

Glue A [φ] e — Glue type formation

Glue U0 [i0] e    -- phi=0: reduces to base type U0
Glue U0 [i1] e    -- phi=1: reduces to dom(e)

Changed from previous versions: The third argument was previously an untyped fibre T. It is now an equivalence e : Equiv T A. The β-rule for φ = 1 now correctly extracts the domain of e.

glue [φ] t a — glue element introduction

Introduces an element of a Glue type. t lives in the domain of the equivalence (the fibre type) and a lives in the base type A. When φ = 1 the result is t; when φ = 0 the result is a.

unglue [φ] e g — unglue element elimination

Projects out of a Glue type, returning an element of the base type.

Changed from previous versions: When φ = 1, unglue previously returned g unchanged. It now correctly applies the equivalence's forward map — equivFwd e g. This is the coherence condition that makes Glue/unglue a proper round-trip.

12 Equivalences and univalence

Equiv A B — equivalence type

The type of equivalences from A to B. Both arguments are atoms. Typing rule: If A : U_n and B : U_n then Equiv A B : U_n.

Equiv U0 U0          -- type of self-equivalences of U0; lives in U1
Π(e : (Equiv U0 U0)). Path U1 U0 U0    -- Pi over an equivalence

mkEquiv A B f g η ε — building an equivalence

ArgTypeRole
AatomDomain type
BatomCodomain type
f( term ) or atomForward map f : A → B
g( term ) or atomBackward map g : B → A
η( term ) or atomLeft homotopy η : Π(a:A). Path A a (g (f a))
ε( term ) or atomRight homotopy ε : Π(b:B). Path B (f (g b)) b
 
-- The identity equivalence on A:
mkEquiv A A (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b)

-- Closed term at A = U0:
mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b)

equivFwd e x — applying the forward map

β-rule: equivFwd (mkEquiv A B f g η ε) x ≡ f x

equivFwd (mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b)) U0
equivFwd mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b) U0  -- ❌ wrong

ua e — the univalence map

Converts an equivalence e : Equiv A B into a path ua e : Path U_n A B. ua itself does not reduce further — its computational content is exposed via transport (uaβ below).

ua (mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b))
-- : Path U1 U0 U0

transport p x — transporting along a path

Given a path p : Path U_n A B and an element x : A, produces transport p x : B.

ConditionReduction
p = ua etransport (ua e) x ≡ equivFwd e x  (uaβ — key univalence computation)
p = ⟨i⟩ A (constant)transport (⟨i⟩ A) x ≡ x

Worked univalence example

-- Step 1: build the identity equivalence on U0
mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b)
-- : Equiv U0 U0

-- Step 2: turn it into a path in U1
ua (mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b))
-- : Path U1 U0 U0

-- Step 3: transport U0 along that path — should give back U0
transport (ua (mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b))) U0
-- ≡  equivFwd (mkEquiv ...) U0   by uaβ
-- ≡  (λx. x) U0                  by equivFwd β
-- ≡  U0                           by λ β

13 Common mistakes

Using a name before it is defined

def myType : myType = U0    -- ❌ myType not yet in scope

def myType = U0
def myAlias : myType = U0   -- ✅ myType is in scope

Forgetting def makes a name globally available

mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b)  -- ❌ not bound

def idEquiv = mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b)
transport (ua idEquiv) U0    -- ✅ idEquiv is in scope

Omitting the type annotation on a bare lambda

def myId = λx. x                       -- ❌ cannot infer type
def myId : Π(x:U0). U0 = λx. x        -- ✅ type given explicitly

Confusing def and check

check myLemma : Path U1 U0 U0 = ua idEquiv   -- OK for testing
myLemma                                        -- ❌ unbound — check didn't bind it

def myLemma : Path U1 U0 U0 = ua idEquiv
myLemma                                        -- ✅ works

Forgetting square brackets around the face

hcomp U1 i0 ⟨i⟩ U0 U0    -- ❌ wrong
hcomp U1 [i0] ⟨i⟩ U0 U0  -- ✅ correct

The face argument of hcomp, Glue, glue, and unglue is always written inside […].

Forgetting to parenthesise compound arguments

Path U2 Path U1 U0 U0 Path U1 U0 U0           -- ❌ parser fails
Path U2 (Path U1 U0 U0) (Path U1 U0 U0)       -- ✅ correct

Confusing interval literals with names

i0      -- interval literal, type 𝕀
i0foo   -- looked up as a name — make sure it's in scope

Wrong capitalisation of keywords

Glue     -- type-formation keyword (capital G)
glue     -- element-introduction keyword (lowercase g)
Equiv    -- type keyword (capital E)
equivFwd -- elimination keyword (camelCase)

Nested Π without re-wrapping

Π(A:U0). Π(x:A). A    -- ✅ each Π has its own (x:A). part
Π(A:U0)(x:A). A       -- ❌ not valid syntax

Forgetting that mkEquiv needs six arguments

mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a) (λb. ⟨i⟩ b)
--       A    B    f             g             η              ε

mkEquiv U0 U0 (λx. x) (λx. x) (λa. ⟨i⟩ a)   -- ❌ missing ε

When in doubt, add parentheses — they never hurt and the inner term always gets the full grammar.
